# Appointment Assistent Utility-Based Multi-Agent Platform

This project demonstrates an LLM-powered healthcare appointment agent that combines:

- executable planning;
- deterministic utility scoring;
- a **pre-execution plan critic**;
- a **post-execution recommendation critic**;
- bounded planner revision;
- mandatory human confirmation before booking.

The two critics have different responsibilities and run at different stages.

## Architecture

```text
Natural-language request
        |
        v
PlannerAgent
  extracts goal/preferences
  creates business workflow
        |
        v
PlanCriticAgent
  reviews the plan before execution
        |
        +-- rejected --> PlannerAgent.revise_plan(..., rejection_stage="plan")
        |                 no tools have run yet
        |
        v
Business action executor
  find_appointments
        |
        v
  evaluate_options
        |
        v
  rank_options
        |
        v
Recommendation CriticAgent
  validates the produced recommendation
        |
        +-- rejected --> PlannerAgent.revise_plan(..., rejection_stage="recommendation")
        |                 next attempt starts from a revised plan
        |
        v
Human confirmation request
        |
        v
Booking endpoint
```

## Why there are two critics

### 1. Plan critic: before execution

`PlanCriticAgent` reviews the workflow generated by the planner before any appointment tool or utility action runs.

It checks that:

- the plan addresses the user's goal;
- only supported business actions are used;
- step IDs and dependencies are valid;
- required producer steps run before their consumers;
- the workflow can produce a recommendation;
- hard constraints and soft preferences are interpreted correctly;
- the plan does not contain booking, critic-review, confirmation, or invented actions.

The orchestrator also performs deterministic structural validation. Structural failures cannot be overridden by the LLM critic.

If the plan critic rejects the plan, the planner receives the issues and creates a revision. No business step is executed before approval.

### 2. Recommendation critic: after execution

`CriticAgent` runs only after the approved business plan has produced:

- candidate appointments;
- deterministic utility scores;
- a selected winner;
- a user-facing recommendation.

It checks that:

- the recommended slot exists;
- the recommendation matches the deterministic winner;
- the message accurately describes the selected appointment;
- unmet soft preferences are disclosed truthfully;
- confirmation is requested before booking.

If this critic rejects the result, its feedback is sent back to the planner with `rejection_stage="recommendation"`.

## Planner-generated business plan

The planner creates only business steps. Platform guardrails are not optional planner actions.

Supported planner actions are defined by `PlanAction`:

```text
find_appointments
evaluate_options
rank_options
```

Example plan:

```json
{
  "goal": "Find the best primary care appointment",
  "preferences": {
    "preferred_provider": "Dr. Lee",
    "preferred_specialty": "primary care",
    "preferred_time_of_day": "evening",
    "max_distance_miles": 15
  },
  "steps": [
    {
      "step_id": "search",
      "action": "find_appointments",
      "purpose": "Retrieve matching appointment candidates",
      "depends_on": []
    },
    {
      "step_id": "evaluate",
      "action": "evaluate_options",
      "purpose": "Apply constraints and calculate utility scores",
      "depends_on": ["search"]
    },
    {
      "step_id": "rank",
      "action": "rank_options",
      "purpose": "Create a user-facing recommendation",
      "depends_on": ["evaluate"]
    }
  ],
  "missing_required_information": []
}
```

## Action registry

The orchestrator maps planner-selected symbolic actions to executable handlers:

```python
self.action_handlers = {
    PlanAction.FIND_APPOINTMENTS: self._execute_find_appointments,
    PlanAction.EVALUATE_OPTIONS: self._execute_evaluate_options,
    PlanAction.RANK_OPTIONS: self._execute_rank_options,
}
```

The planner decides the business workflow and dependencies, but it may only select actions the application knows how to execute.

The critic-review and confirmation stages are enforced by the platform and cannot be removed by the planner.

## Retry behavior

The default configuration allows one initial plan plus two revisions:

```python
UtilityHealthcareOrchestrator(max_plan_retries=2)
```

A retry may be triggered at either stage:

```text
Plan critic rejection
    -> revise plan using plan-stage feedback

Recommendation critic rejection
    -> revise plan using recommendation-stage feedback,
       deterministic decision, and rejected recommendation
```

This creates at most three total attempts and prevents infinite planner/critic loops.

## LLM integration

Model calls are isolated in [`app/llm/client.py`](app/llm/client.py):

```python
response = self.client.responses.parse(
    model=self.model,
    input=[
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_prompt},
    ],
    text_format=response_model,
)
```

The following agents reuse that adapter:

- `PlannerAgent` — creates and revises executable business plans.
- `PlanCriticAgent` — validates a plan before execution.
- `LLMUtilityAgent` — explains and ranks deterministically scored candidates.
- `CriticAgent` — validates the resulting recommendation.

The LLM cannot bypass hard constraints or invent the winning appointment. Application code verifies that the selected slot matches the deterministic utility winner.

## Setup

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Set your environment variables in `.env`:

```text
OPENAI_API_KEY=your_api_key_here
OPENAI_MODEL=gpt-4.1-mini
```

Run the server:

```bash
python app.py
```

## Call the agent

```bash
curl -X POST http://127.0.0.1:5050/agent/chat \
  -H 'Content-Type: application/json' \
  -d '{
    "message": "Find me a primary-care appointment with Dr. Lee, preferably in the evening and within 15 miles",
    "include_trace": true
  }'
```

The agent may recommend the best available appointment even when a soft preference is unavailable. It must disclose the mismatch and ask for confirmation.

The `/agent/chat` endpoint does not book the appointment.

## Confirm booking

```bash
curl -X POST http://127.0.0.1:5050/agent/confirm \
  -H 'Content-Type: application/json' \
  -d '{"patient_id":"PATIENT-123", "slot_id":"S1"}'
```

## Deterministic comparison endpoint

`POST /agent/recommend` remains available to demonstrate the difference between a pure deterministic utility service and the LLM-powered multi-agent flow. It does not call OpenAI.

## Tests

```bash
pytest -q
```

Tests use fake LLM implementations, so they do not consume API tokens. The suite includes cases proving that a plan rejected by the plan critic executes no business tools.
